# 2026-03-30 SRE Daily Architecture Log

## 🧩 오늘의 핵심 문제 (Problem)

오늘의 핵심 문제는 **brand-web 애플리케이션의 배포 및 운영 환경을 체계적으로 설계하고, CI/CD 파이프라인을 구축하는 것**이었다.  
기존에는 단순한 프로토타입 또는 로컬 개발 환경에서만 실행 가능했던 `brand-web` 애플리케이션이, **실제 브랜드 웹사이트로의 전환을 위한 구조적 준비**가 필요했다.  

구체적으로 다음과 같은 문제들이 있었다:

1. **배포 자동화 누락**: `brand-web` 애플리케이션이 GitHub Pages로 자동 배포되는 파이프라인이 구성되지 않았다.
2. **환경 설정 불일치**: `.gitignore`와 `next.config.ts` 등의 설정이 브랜드 웹에 맞게 조정되지 않아, 개발 및 배포 시 오류가 발생할 가능성이 있었다.
3. **코드 품질 및 유지보수성 부족**: ESLint 설정이 누락되었고, `@AGENTS.md`와 같은 AI 협업을 위한 문서가 추가되어야 했다.

## 🏗️ 해결 방안 및 시스템 아키텍처 개선

### 1. GitHub Actions 기반 자동 배포 파이프라인 구축  
`e75cfb739d41c1d2f2f209c329a9c2c162767226` 커밋을 통해, GitHub Pages에 배포할 수 있도록 **GitHub Actions 워크플로우**를 구성했다.  

- `deploy-brand-web.yml`은 다음과 같은 핵심 단계로 구성되어 있다:
  - `Checkout`: 소스코드를 가져온다.
  - `Setup Node`: Node.js 환경을 설정하고 `npm ci`로 의존성을 설치한다.
  - `Build with Next.js`: `next build` 명령어로 빌드한다.
  - `Upload artifact`: 빌드 결과를 GitHub Pages에 업로드할 수 있도록 아티팩트로 저장한다.
  - `Deploy to GitHub Pages`: GitHub Pages에 배포한다.

📌 **아키텍처적 이점**:
- **분리된 배포 파이프라인**: `brand-web` 애플리케이션은 `main` 브랜치에 푸시될 때 자동으로 배포되며, 별도의 CI/CD 파이프라인을 통해 운영과 개발의 경계를 명확히 한다.
- **확장 가능성**: 향후 `staging`, `production` 환경에 따라 워크플로우를 분리하거나, 릴리스 관리를 위한 브랜치 전략을 추가할 수 있다.

---

### 2. 브랜드 웹 애플리케이션의 기본 설정 및 환경 정리  
`1ba8e6f03f05d317a90fe8013e449e6ca14804b5` 커밋에서는 `brand-web` 애플리케이션의 초기 설정 파일들을 추가했다.

#### 📁 `.gitignore` 구성
- `node_modules`, `.next`, `out`, `build` 등 빌드 및 의존성 폴더를 무시하도록 구성
- `*.env*`, `.DS_Store` 등 민감한 정보 및 운영 환경에서 불필요한 파일을 제외

📌 **아키텍처적 이점**:
- **보안 강화**: 민감한 정보나 빌드 결과가 git에 커밋되지 않도록 하여, 보안 위험을 줄인다.
- **Git 트래픽 최적화**: 불필요한 파일이 커밋되지 않아, 저장소의 크기와 성능 향상에 기여한다.

#### 🧱 `next.config.ts` 설정
```ts
const nextConfig: NextConfig = {
  output: 'export',
};
```

📌 **아키텍처적 이점**:
- `output: 'export'` 설정은 **Static Site Generation (SSG)** 방식으로 빌드되도록 하여, **서버리스 배포**에 최적화된 구조로 전환
- GitHub Pages와 같은 정적 호스팅 환경에서 동작 가능하며, **서버 부담 감소 및 성능 향상**을 기대할 수 있다.

#### 📄 `@AGENTS.md` 및 `CLAUDE.md` 문서 추가
- `@AGENTS.md`는 AI 협업 시 필요한 Next.js 버전의 특이사항을 안내
- `CLAUDE.md`는 `@AGENTS.md`를 참조하도록 구성

📌 **아키텍처적 이점**:
- **개발자 협업 및 자동화 준비**: AI 기반 개발 시스템(예: Claude)과의 호환성을 위한 문서화로, 향후 자동 코드 생성 시스템에 대한 안정성을 제공한다.

## 🧠 오늘의 아키텍처적 통찰 (Architecture Insight)

### 1. **정적 배포 전략의 시스템적 가치**
- `brand-web` 애플리케이션은 `output: 'export'`로 정적 생성 방식으로 설계되었다. 이는 **서버리스 아키텍처의 핵심 원칙** 중 하나인 "무 상태"와 잘 맞아떨어진다.
- GitHub Pages와 같은 정적 호스팅 서비스를 활용함으로써, **서버 운영 비용과 복잡성의 감소**를 실현할 수 있다.

### 2. **CI/CD 파이프라인의 자동화 및 확장 가능성**
- GitHub Actions를 통해 `main` 브랜치에 대한 자동 배포가 가능해졌으며, 이는 **GitOps 방식의 기반**이 된다.
- 향후 `staging`, `production` 브랜치를 분리하거나, 배포 전/후 테스트를 추가하는 것도 쉽게 가능하다.

### 3. **환경 분리 및 코드 품질 보장**
- `.gitignore`와 `eslint.config.mjs` 등의 설정은 **개발 환경과 운영 환경의 분리를 강화**하고, 코드 품질 유지에 기여한다.
- `eslint-config-next`를 활용한 설정은 **코드 스타일의 일관성**과 **버그 예방**에 기여한다.

## 🧭 결론

오늘의 작업은 단순한 브랜드 웹사이트 생성이 아닌, **전체 시스템의 배포 및 운영 체계에 대한 설계**를 의미한다.  
`brand-web` 애플리케이션은 단순한 정적 웹사이트로 시작했지만, GitHub Pages와 CI/CD 파이프라인을 통해 **확장 가능한 시스템 아키텍처**로 진화했다.  

이러한 변화는 **지속적인 개발과 운영의 효율성**, 그리고 **브랜드와 시스템 간의 일관성 유지**를 위해 매우 중요한 단계이다.  
향후에는 이 구조를 기반으로 **다양한 브랜드 웹사이트의 자동화 배포 시스템**으로 확장할 수 있을 것이다.

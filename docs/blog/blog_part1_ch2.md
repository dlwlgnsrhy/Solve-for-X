# [엔지니어링 편] 2장. 설계의 재구성: 두뇌(Hermes)와 손발(OpenAgent)
**부제: 명령을 내리는 자와 코드를 짜는 자를 철저히 분리하라**

1장에서 살펴본 바와 같이, "네가 알아서 다 해봐"라는 식의 단일 에이전트(Monolithic Agent) 접근 방식은 무한 루프와 좀비 프로세스라는 끔찍한 결과를 낳았습니다. 에이전트가 코드를 짜다가 에러에 갇혀버리면, 전체 워크플로우를 관장하는 메인 프로세스마저 함께 뻗어버렸기 때문입니다.

더 이상 제 소중한 Mac이 비명을 지르게 둘 수는 없었습니다. 저는 단일 '에이전트'가 아닌 견고한 '시스템'을 구축하기 위해 아키텍처를 근본적으로 뜯어고치기로 결심했습니다. 

핵심은 단 하나, **'책임의 철저한 분리(Separation of Concerns)'**였습니다.

---

### 🧠 두뇌와 손발의 분리: Hermes & OpenAgent

만약 인간 개발팀이라면, 프로젝트 매니저(PM)가 전체 일정을 관리하고, 주니어 개발자가 실제로 코드를 작성합니다. 주니어 개발자가 컴파일 에러를 낸다고 해서 PM이 심정지를 일으키지는 않죠. 하지만 기존의 제 자동화 스크립트는 이 둘이 하나의 프로세스 안에 묶여 있었습니다.

그래서 저는 역할을 완벽하게 두 개로 쪼갰습니다.

1. **오케스트레이터, Hermes (두뇌)**
   Hermes는 전체 워크플로우를 관장하는 매니저입니다. 직접 코드를 짜거나 빌드 명령어를 치지 않습니다. 대신 작업의 우선순위를 정하고, 상태를 추적하며, 텔레그램을 통해 사용자와 소통합니다. 가장 중요한 역할은 하위 작업자(Worker)가 사고를 치지 않는지 감시하고, 작업 시간을 초과하면 가차 없이 프로세스를 차단하는 것입니다.

2. **지능형 작업자, OpenAgent (손발)**
   OpenAgent는 철저히 통제된 환경에서 코드를 생성하고 스캐폴딩(Scaffolding)하는 데 특화된 실무자입니다. Hermes로부터 "Imjong Care 앱의 홈 화면 Riverpod 상태 관리를 구현해"라는 명확하고 제한된 미션만을 부여받습니다. 코드를 짜고, 테스트를 돌리고, 결과를 Hermes에게 보고하는 역할만 수행합니다.

이렇게 분리해두니, OpenAgent가 코딩 중에 무한 루프에 빠지거나 크래시가 나더라도 두뇌 역할을 하는 Hermes는 여전히 무사히 살아있어 텔레그램으로 저에게 알림을 보낼 수 있게 되었습니다.

### 🔌 단일 거대 모델(Qwen 35B)로의 대통합

역할을 분리한 후 직면한 또 다른 문제는 '지능의 파편화'였습니다. 예전에는 작업별로 가벼운 모델과 무거운 모델을 섞어 쓰다 보니, Hermes가 내린 고차원적인 지시를 하위 에이전트가 제대로 이해하지 못하는 병목 현상이 생겼습니다.

저는 과감하게 모든 에이전트의 뇌를 하나로 통일했습니다. 바로 로컬/프록시 환경에서 돌아가는 강력한 **`Qwen3.6-35B-A3B-FP8`** 모델이었습니다. 

시스템 전반의 `.env.shared` 설정을 뜯어고치고 프록시 엔드포인트(포트 30008)를 하나로 맞춰, Hermes든 OpenAgent든 SRE 봇이든 모두 동일한 35B 파라미터의 강력한 추론 능력을 공유하도록 아키텍처를 대통합했습니다. 하나의 거대한 지능이 '지시'와 '실행' 양쪽을 모두 담당하게 되니, 컨텍스트 유실 없이 마치 한 사람이 생각하고 움직이는 것처럼 매끄러운 파이프라인이 완성되었습니다.

### ⚙️ 완벽한 자율 시스템의 실제 작동 방식

이제 제 Mac에서는 버그가 발생하면 다음과 같은 일이 벌어집니다.

1. **Trigger:** 제가 텔레그램으로 "Legacy Vault 앱의 로딩 버그 좀 고쳐줘"라고 툭 던집니다.
2. **Planning (Hermes):** Hermes가 메시지를 수신하고, 어떤 파일을 수정해야 할지 계획을 세운 뒤 OpenAgent에게 작업 지시서(Ticket)를 발급합니다.
3. **Execution (OpenAgent):** OpenAgent가 코드를 수정하고 로컬에서 빌드 테스트를 돌립니다.
4. **Supervision:** 만약 OpenAgent가 에러를 3번 이상 반복하며 헤매면? 지켜보던 Hermes가 즉각 개입하여 OpenAgent의 작업을 강제 중단(Clean Exit)시키고 텔레그램으로 저에게 실패 로그와 함께 도움을 요청합니다.

드디어 '통제 불능의 블랙박스'가 '예측 가능한 무인 개발팀'으로 진화하는 순간이었습니다.

저는 드디어 이겼다고 생각했습니다. Hermes가 처음으로 무한 루프를 감지하고, "Process Terminated" 로그를 띄운 뒤 제게 텔레그램 알림을 보냈을 때, 마치 완벽한 SRE 시스템을 구축한 천재가 된 것 같았죠.

하지만 다음 날 아침, 제 Mac은 완전히 먹통이 되어 있었습니다. 팬(Fan)은 비명을 지르고 있었고요.

활동 활성 상태 보기를 열어보고 경악했습니다. 분명 Hermes는 OpenAgent의 메인 Python 스크립트를 성공적으로 죽였습니다. 하지만 OpenAgent가 테스트를 위해 띄워둔 자식 프로세스들—Webpack 개발 서버, 응답 없는 Dart 분석기, 헤드리스 Node.js 인스턴스들—은 메인 프로세스와 연결이 끊어진 채 고아(Orphan) 상태로 남아있었습니다. Hermes가 머리는 잘라냈지만, 몸통은 백그라운드에 남아 제 Mac의 메모리를 파먹고 있었던 것입니다.

단순한 논리적 종료(Logical Termination)만으로는 부족했습니다. OS 레벨에서의 완벽한 "Clean Exit"이 필요했습니다.

이 문제를 해결하기 위해 저는 운영체제의 어두운 영역인 프로세스 트리(Process Tree)와 POSIX 시그널, 그리고 세션 ID(Session ID)의 세계로 파고들어야 했습니다. 좀비처럼 살아나는 이 'AI 프로세스'들을 어떻게 시스템 레벨에서 완벽하게 소탕했는지, 다음 3장에서 본격적인 하드코어 트러블슈팅을 다루어 보겠습니다.

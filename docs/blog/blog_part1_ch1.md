# [엔지니어링 편] 1장. "알아서 코딩해 줘"의 숨겨진 함정
**부제: 완전 자율 AI 에이전트가 내 Mac에서 일으킨 참사들**

"앱 기획서 여기 있어. 아키텍처는 Clean Architecture로 잡고, 알아서 코딩하고 빌드까지 해줘."

최근 1~2년 사이 거대 언어 모델(LLM)과 자율 에이전트(Autonomous Agent) 생태계가 폭발적으로 성장하면서, 수많은 개발자들이 위와 같은 달콤한 상상을 했습니다. 저 역시 예외는 아니었습니다. 'Living Well = Dying Well'이라는 거대한 철학을 담은 3종의 생애 주기 앱(Imjong Care, Memento Mori, Legacy Vault)을 혼자서 모두 개발해야 했기 때문이죠.

제 목표는 명확했습니다. 제가 잠든 사이에도, 제 Mac(로컬 환경)에서 AI 에이전트가 쉴 새 없이 코드를 짜고, 버그를 잡고, 앱을 빌드하는 **'완전 자율 무인(Hands-off) 개발팀'**을 꾸리는 것이었습니다. 

하지만 호기롭게 로컬 LLM(Qwen 35B)과 에이전트 코드를 실행하고 맞닥뜨린 현실은, 우아한 자동화가 아니라 그야말로 **'카오스(Chaos)'** 그 자체였습니다.

---

### 🚨 참사 1. 무한 루프(Infinite Loop)의 늪

에이전트에게 자율성을 부여한다는 것은, 에이전트 스스로 터미널 명령어를 실행하고 에러 로그를 읽고 코드를 수정할 권한을 준다는 뜻입니다. 문제는 AI가 가끔 '환각(Hallucination)'에 빠지거나 컨텍스트를 잃어버릴 때 발생합니다.

*   에이전트가 존재하지 않는 패키지를 설치하려고 시도합니다. (`npm install 환각_패키지`)
*   당연히 에러가 납니다.
*   에이전트는 에러 로그를 읽고, "아, 오타가 있었구나"라며 똑같은 잘못된 패키지명을 다시 설치 시도합니다.

이 과정이 백그라운드에서 수백, 수천 번 반복됩니다. 제가 모니터를 다시 켰을 때는 터미널 스크롤이 끝도 없이 올라가고 있고, LLM 프록시 API 호출 비용(혹은 로컬 GPU 자원)이 허공으로 증발하고 있었습니다. 

### 🧟 참사 2. 메모리를 갉아먹는 AI 좀비 프로세스

에이전트가 코드를 작성하고 테스트하기 위해서는 필연적으로 Node.js 서버를 띄우거나, Python 백그라운드 스크립트를 실행해야 합니다. 만약 테스트가 실패하거나 에이전트가 비정상 종료(Crash)되면 어떻게 될까요?

에이전트 프로세스는 죽었지만, **에이전트가 띄워놓은 자식 프로세스(서버, 빌드 스크립트 등)는 고아(Orphan) 상태가 되어 백그라운드에 그대로 남습니다.** 

며칠 동안 에이전트 자동화를 돌려놓고 맥북이 이륙할 듯이 팬(Fan)이 도는 소리에 활동 활성 상태 보기(Activity Monitor)를 열어본 적이 있습니다. 정체를 알 수 없는 Node와 Python 데몬 수십 개가 제 Mac의 RAM을 99%까지 집어삼키고 있었습니다. 저는 이것을 **'AI 좀비 프로세스'**라고 부르기 시작했습니다.

### 🕳️ 참사 3. 통제 불능의 블랙박스와 데드락(Deadlock)

이러한 문제들을 해결하기 위해 에이전트의 상태를 텔레그램 메신저로 받아보도록(Feedback Bridge) 연동했습니다. 버그가 터지면 텔레그램으로 알려주고, 제가 승인하면 고치는 식이었죠.

하지만 비동기 네트워크 환경은 또 다른 지옥이었습니다.
*   에이전트가 텔레그램 API로 메시지를 보내려다 네트워크 타임아웃이 발생합니다.
*   에러 핸들링이 제대로 되어 있지 않은 에이전트는 무한 대기(Deadlock) 상태에 빠집니다.
*   저는 텔레그램으로 아무 메시지도 오지 않으니 "작업을 잘하고 있나 보다"라고 착각합니다.

실제로는 모든 프로세스가 멈춰서 허공만 바라보고 있는 통제 불능의 블랙박스 상태였던 것입니다.

---

### 💡 결론: AI에게 필요한 건 '지능'이 아니라 '목줄(SRE)'이다.

수많은 좀비 프로세스를 `kill -9`로 죽이고, 멈춰버린 데몬을 재시작하면서 저는 한 가지 뼈저린 깨달음을 얻었습니다.

> "완전 자율 개발 시스템을 만드는 데 있어 진짜 어려운 기술은 '얼마나 똑똑한 AI를 쓰느냐'가 아니다. 폭주하는 AI를 통제하고, 실패를 복구하며, 깔끔하게 프로세스를 정리하는 **SRE(사이트 신뢰성 엔지니어링) 인프라**를 구축하는 것이다."

단순히 "알아서 해줘"라는 프롬프트만으로는 프로덕션 레벨의 파이프라인을 만들 수 없었습니다. 에이전트가 사고를 치지 않도록 가이드라인을 세우고, 작업이 끝나면 흔적 없이 메모리에서 사라지게 만들며, 문제가 생기면 즉시 알림을 보내는 견고한 '오케스트레이션(Orchestration)'이 필요했습니다.

그래서 저는 거대하고 똑똑하지만 통제가 안 되는 하나의 AI 대신, 시스템을 통제하는 **'두뇌'**와 실제로 코드를 짜는 **'손발'**을 철저하게 분리하기로 결심합니다. 

이것이 바로 제 Mac에 평화를 가져다준 **Hermes(오케스트레이터)**와 **OpenAgent(작업자)** 아키텍처의 시작이었습니다. *(다음 장에서 계속됩니다...)*

# [엔지니어링 편] 3장. 내 Mac에서 자라나는 AI 좀비 소탕 작전
**부제: 세션 ID(Session ID)를 활용한 완벽한 Clean Exit 전략**

2장에서 언급했듯, 오케스트레이터(Hermes)와 작업자(OpenAgent)를 분리하여 논리적인 제어권을 확보했을 때 저는 모든 문제가 해결된 줄 알았습니다. 하지만 다음 날 아침, 비명을 지르는 Mac의 팬(Fan) 소리와 99%를 꽉 채운 RAM 사용량은 저에게 운영체제(OS)의 차가운 현실을 가르쳐주었습니다. 

Hermes는 분명 에러의 무한 루프를 감지하고 OpenAgent의 메인 스크립트를 성공적으로 종료시켰습니다. 그런데 왜 메모리는 줄어들지 않았을까요?

### 🧟 고아(Orphan)가 되어버린 AI의 파편들

문제를 이해하려면 Linux/macOS(POSIX) 환경의 프로세스 트리(Process Tree) 구조를 알아야 합니다. 

OpenAgent가 코드를 테스트하기 위해 터미널 명령어(예: `npm run dev` 또는 `flutter test`)를 실행하면, 메인 Python 스크립트 아래에 여러 자식 프로세스(Subprocess)가 생성됩니다. 

만약 Hermes가 타임아웃을 선언하고 OpenAgent의 메인 스크립트를 단순히 `kill` 명령어로 죽여버리면 어떻게 될까요? 메인 프로세스는 사라지지만, 이미 실행 중이던 Webpack 빌드 서버나 Dart 분석기 같은 자식 프로세스들은 종료되지 않고 **고아(Orphan) 프로세스**가 됩니다. 이들은 macOS의 최상위 프로세스인 `launchd` 밑으로 편입되어, 누구의 통제도 받지 않은 채 백그라운드에서 영원히 RAM과 CPU를 갉아먹습니다.

저는 이것을 **'AI 좀비 프로세스'**라고 불렀습니다. 매번 자동화 루프가 실패할 때마다 제 Mac에는 새로운 좀비들이 쌓여가고 있었습니다.

### 🚫 실패한 접근법: 이름으로 찾아 죽이기 (grep & pkill)

처음에는 무식한 방법을 썼습니다. Hermes가 OpenAgent를 죽일 때, `pkill -f node`나 `pkill -f dart` 같은 명령어를 함께 실행하도록 한 것이죠.

결과는 처참했습니다. 에이전트가 띄운 서버뿐만 아니라, **제가 직접 작업하고 있던 다른 프로젝트의 로컬 서버까지 모조리 죽어버렸기 때문입니다.** 제 Mac은 여러 앱(Imjong Care, Legacy Vault 등)을 동시에 개발하는 환경이었기에, 이름만으로 프로세스를 죽이는 것은 지뢰밭을 걷는 것과 같았습니다.

내 에이전트가 생성한 프로세스만 정확하게 타겟팅하여 일망타진할 방법이 필요했습니다.

### 🎯 해결책: 프로세스 그룹(PGID)과 세션(Session)의 분리

운영체제의 깊은 곳을 파고들다 찾아낸 해답은 바로 **세션 ID(Session ID)**와 **프로세스 그룹(Process Group)**이었습니다.

Hermes가 OpenAgent를 실행할 때, 단순히 하위 프로세스로 실행하는 것이 아니라 `os.setsid()`(Python 기준)를 호출하여 **새로운 프로세스 세션**을 부여하도록 아키텍처를 수정했습니다. 

이렇게 하면 OpenAgent와 그 밑에서 파생되는 수십 개의 자식 프로세스(Node, Dart, Python 등)가 모두 동일한 프로세스 그룹 ID(PGID)를 공유하게 됩니다. 마치 하나의 캡슐 안에 모든 작업물을 가두는 것과 같습니다.

이제 Hermes가 OpenAgent를 중단시켜야 할 때, 개별 프로세스의 PID를 찾을 필요가 없습니다. 대신 프로세스 그룹 전체에 종료 시그널을 보냅니다.
`os.killpg(pgid, signal.SIGTERM)`

이 명령어 한 줄이 마법을 부렸습니다. 메인 스크립트는 물론이고, 깊숙한 곳에서 숨어 돌고 있던 헤드리스 브라우저나 데몬들까지, **해당 세션에 속한 모든 좀비 프로세스들이 단 한 번의 명령으로 완벽하게 소멸**되었습니다.

### 🧹 진정한 의미의 Clean Exit

프로세스 그룹을 날려버리는 것에 더해, 작업 중 생성된 임시 파일(Temporary files)과 잠겨버린 데이터베이스 Lock을 해제하는 정리(Cleanup) 로직까지 추가했습니다.

이제 제 자동화 워크플로우는 다음과 같이 동작합니다.
1. OpenAgent가 무한 루프에 빠집니다.
2. Hermes가 타임아웃을 감지합니다.
3. Hermes가 OpenAgent의 프로세스 그룹(PGID) 전체에 SIGTERM을 날려 모든 자식 프로세스(좀비)를 즉각 소탕합니다.
4. 작업 폴더의 임시 찌꺼기들을 삭제합니다.
5. 완전히 깨끗해진 상태(Clean Exit)를 확인한 후, 비로소 텔레그램으로 "작업이 안전하게 중단되었습니다"라는 알림을 보냅니다.

이제 제 Mac은 아무리 며칠 동안 자동화를 돌려놓아도 메모리 누수 없이 쾌적한 상태를 유지합니다. 좀비 프로세스의 공포에서 완벽하게 해방된 것이죠.

하지만, 시스템 레벨의 안정성을 확보하고 나니 곧바로 **네트워크 레벨의 재앙**이 찾아왔습니다. 바로 이 모든 것을 보고받는 통로인 '텔레그램 API' 자체가 먹통이 되어 전체 시스템이 멈춰버리는 데드락(Deadlock) 현상이었습니다.

비동기 네트워크 통신에서 발생하는 타임아웃과 데드락을 어떻게 정복했는지, 다음 4장에서 다루어 보겠습니다.

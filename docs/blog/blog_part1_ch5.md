# [엔지니어링 편] 5장. 마침내 완성된 '텔레그램 기반 피드백 브릿지'
**부제: 메신저 하나로 굴러가는 1인 개발팀의 완성**

길고 험난했던 엔지니어링 여정이었습니다. 1장과 2장에서는 통제 불능의 AI가 만들어내는 무한 루프를 막기 위해 아키텍처를 분리(Hermes와 OpenAgent)했고, 3장에서는 OS 레벨의 세션(PGID) 관리를 도입해 좀비 프로세스를 소탕했습니다. 그리고 4장에서는 지수 백오프와 비동기 데몬을 통해 텔레그램 네트워크 데드락까지 완벽하게 잡아냈습니다.

버그, 메모리 누수, 통신 장애. 이 모든 3중고(三重苦)의 SRE(사이트 신뢰성 엔지니어링) 트러블슈팅을 거친 끝에, 마침내 제 Mac에는 절대 멈추지 않는 완벽한 '자율 무인(Hands-off) 개발팀'이 완성되었습니다. 

이번 마지막 장에서는 이 모든 시스템이 결합된 최종 워크플로우, **'피드백 브릿지(Feedback Bridge)'**가 실제로 어떻게 작동하는지 시연해 보겠습니다.

---

### 📱 메신저 하나로 끝나는 엔드투엔드(E2E) 워크플로우

이제 제가 `Imjong Care`나 `Legacy Vault` 앱을 개발하는 방식은 과거와 완전히 다릅니다. IDE를 켜고 키보드를 두드리는 시간보다, 산책을 하며 텔레그램 메신저를 확인하는 시간이 더 길어졌습니다.

최종 워크플로우는 다음과 같이 물 흐르듯 진행됩니다.

**1. 명령 (The Trigger)**
저는 길을 걷다가 문득 앱의 개선점이 떠오르면, 스마트폰을 꺼내 텔레그램 봇에게 메시지를 보냅니다.
> `/task Legacy Vault 앱의 비밀번호 암호화 모듈에 로깅 기능 추가해 줘.`

**2. 수신 및 계획 (Hermes)**
Mac에서 백그라운드로 대기 중이던 오케스트레이터 'Hermes'가 메시지를 수신합니다. 단일 통일된 `Qwen3.6-35B` 모델을 활용하여 이 작업에 필요한 파일이 무엇인지 분석하고, 작업 지시서를 작성합니다.

**3. 격리된 실행 (OpenAgent)**
Hermes는 작업자 'OpenAgent'에게 지시서를 넘기며 새로운 프로세스 세션(Session ID)을 생성하여 실행시킵니다. OpenAgent는 코드를 수정하고, 로컬에서 Python이나 Flutter 빌드 테스트를 돌립니다.
만약 에러가 발생해서 루프에 빠지려 하면? Hermes가 설정된 타임아웃 규칙에 따라 가차 없이 프로세스 그룹(PGID) 전체를 날려버리고(Clean Exit) 다시 시도합니다. 제 Mac의 메모리는 단 1byte도 낭비되지 않습니다.

**4. 비동기 보고 (Worker Monitor)**
작업이 무사히(혹은 실패로) 끝나면 OpenAgent는 최종 결과를 로컬 파일 시스템에 저장하고 깔끔하게 소멸합니다. 
이를 지켜보던 독립 데몬(`worker_monitor.py`)이 파일을 감지하고, 네트워크 타임아웃에 대비한 견고한 재시도(Retry) 로직을 거쳐 제 텔레그램으로 시각화된 최종 리포트와 Git Diff를 전송합니다.

**5. 검토 및 승인 (Human-in-the-loop)**
커피를 마시던 제 폰에 알림이 울립니다.
> 🔔 **작업 완료 보고서:** 로깅 기능 추가 완료. 
> 수정한 파일 내역과 테스트 결과를 확인하시겠습니까? (승인 / 거절)

제가 '승인' 버튼을 누르면 코드가 병합(Merge)되고, Github Actions를 통해 실제 프로덕션 서버와 모바일 앱으로 자동 배포(CI/CD)됩니다.

### 🌉 마무리: 우리는 왜 이토록 '로컬(Local)'에 집착하는가?

메신저로 버그를 던지면 알아서 고치고 보고하는 완벽한 파이프라인. 이로써 길었던 [엔지니어링 편] 시리즈가 막을 내렸습니다.

하지만 여기까지 글을 읽으신 분들이라면, 마음속에 강렬한 의문이 하나 남으실 겁니다.

> "결과물은 대단하지만... 굳이 왜 이렇게까지 힘들게 만들었지? 그냥 ChatGPT API나 Claude API를 연동해서 쓰면 이런 복잡한 로컬 모델 세팅이나 인프라 고생 없이 훨씬 빠르고 쉽게 만들 수 있었을 텐데?"

맞습니다. 외부 빅테크 기업의 클라우드 API를 쓰면 훨씬 쉬웠을 것입니다. 그럼에도 불구하고 제가 제 Mac이라는 **'로컬(Local)'** 환경에서, 무거운 `Qwen 35B` 모델을 억지로 돌려가며 이 폐쇄적이고 독립적인 인프라를 끝끝내 구축한 데에는 아주 깊은 이유가 있습니다.

이것은 단순히 개발의 편의성 문제가 아닙니다. 이것은 '데이터 주권'과 '인간의 존엄', 그리고 제 모든 프로젝트를 관통하는 **'Living Well = Dying Well'**이라는 철학의 핵심적인 방어선이기 때문입니다.

기술이 인간을 소외시키고 지능이 공기처럼 흔해지는 특이점의 시대. 
저는 왜 거대 빅테크의 클라우드를 거부하고 내 방 구석의 로컬 서버를 선택했을까요?

이어지는 **[Part 2. 기술 철학 편 - 특이점의 시대, 인간의 존엄은 '로컬'에서 시작된다]**에서 그 소름 돋는 진짜 이유를 공개하겠습니다.

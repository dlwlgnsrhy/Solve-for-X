# [엔지니어링 편] 4장. 텔레그램 데드락(Deadlock)과 비동기 통신 장애 극복
**부제: 에이전트의 입과 귀가 멀어버렸을 때 벌어지는 일들**

3장에서 세션 ID(Session ID)를 도입하여 AI 좀비 프로세스들을 완벽하게 소탕했을 때, 저는 드디어 모든 인프라 작업이 끝났다고 믿었습니다. 프로세스 격리, 메모리 누수 방지, 타임아웃 강제 종료까지 완벽한 SRE(사이트 신뢰성 엔지니어링) 환경이 갖춰졌으니까요.

하지만 시스템이 안정화되자, 그동안 가려져 있던 가장 치명적인 약점이 수면 위로 떠올랐습니다. 바로 **'통신'**이었습니다.

### 🔇 에이전트의 입이 막히다: 네트워크 병목 현상

제 자율 개발팀의 핵심은 '피드백 브릿지(Feedback Bridge)'입니다. Hermes(오케스트레이터)가 작업을 마치거나 치명적인 에러를 발견하면, 저에게 텔레그램(Telegram) 메신저로 보고서를 보냅니다. 저는 밖에서 커피를 마시면서 스마트폰으로 결과를 확인하고 다음 지시를 내리죠.

문제는 네트워크가 언제나 완벽하지 않다는 점입니다.

에이전트가 무거운 코드 파일이나 수백 줄의 에러 로그를 텔레그램 API로 전송하려고 시도합니다. 이때 일시적인 네트워크 불안정, 혹은 텔레그램 서버의 Rate Limit(호출 제한)에 걸려 응답이 지연됩니다. 

만약 파이썬 코드에서 `requests.post()`를 호출할 때 명시적인 `timeout`을 설정하지 않았거나, 에러 핸들링이 허술하다면 어떻게 될까요? 코드는 텔레그램 서버의 응답을 영원히 기다리게 됩니다. 프로세스가 죽은 것도, 에러를 뱉은 것도 아닌, 그저 허공을 바라보며 멈춰버린 **무한 대기(Deadlock)** 상태에 빠지는 것입니다.

### ⏳ 착각의 늪: 일하는 줄 알았는데 기절해 있었다

이 데드락이 끔찍한 이유는 외부에서 알아차리기 매우 어렵다는 점입니다.

밖에서 텔레그램 알림을 기다리던 저는 이렇게 생각합니다.
"오, 이번 빌드는 꽤 오래 걸리네. 에이전트가 열심히 코드를 분석하고 있나 보다!"

하지만 집에 돌아와 Mac의 모니터를 켜보면 진실은 다릅니다. 에이전트는 이미 3시간 전에 모든 빌드와 테스트를 성공적으로 마쳤습니다. 단지 마지막 줄인 `send_telegram_message("성공했습니다")` 함수 안에서 네트워크 응답을 기다리며 3시간째 기절해 있었던 것입니다. 

OS 레벨의 타임아웃(3장)은 작업자(OpenAgent)의 무한 루프를 막아주었지만, 오케스트레이터(Hermes) 자체가 텔레그램 API에 발목이 잡혀 멈춰버리는 것은 막지 못했습니다.

### 🛡️ 해결책 1: 견고한 재시도(Retry) 메커니즘과 타임아웃 강제

가장 먼저 네트워크 요청 계층을 완전히 뜯어고쳤습니다. `telegram_client.py`의 모든 API 호출에 엄격한 타임아웃을 강제했습니다.

단순히 타임아웃만 주면 일시적인 통신 장애 시 보고서가 영영 유실되므로, **지수 백오프(Exponential Backoff)** 알고리즘을 적용한 재시도 로직을 도입했습니다.
*   1차 전송 실패 ➡️ 2초 대기 후 재시도
*   2차 전송 실패 ➡️ 4초 대기 후 재시도
*   3차 전송 실패 ➡️ 8초 대기 후 재시도
*   최종 실패 시 ➡️ 무한 대기에 빠지지 않고 에러를 로컬 로그 파일에 기록한 뒤, 프로세스를 정상적으로 종료(Clean Exit).

이제 네트워크가 끊기더라도 시스템이 데드락에 빠지는 일은 사라졌습니다.

### 🚀 해결책 2: 상태의 강제 추출과 비동기 보고 파이프라인

한 걸음 더 나아가, 아키텍처의 의존성을 분리했습니다. 에이전트가 텔레그램 메시지를 전송하는 행위 자체에 얽매이면 안 됩니다.

에이전트(Hermes & OpenAgent)는 작업이 끝나면 최종 리포트를 메모리가 아닌 로컬 파일 시스템(예: JSON 상태 파일이나 SQLite)에 기록하고 즉시 자신의 임무를 종료합니다. 
그리고 이 상태 파일을 감시하는 가벼운 독립 데몬(`worker_monitor.py`)이 파일이 생성된 것을 감지하면, 에이전트와는 별개의 프로세스에서 텔레그램으로 전송을 담당합니다.

이렇게 **'작업 실행'과 '결과 보고'를 비동기적으로 분리**하자, 텔레그램 서버가 다운되더라도 에이전트의 작업 루프는 아무런 영향을 받지 않고 다음 작업을 이어갈 수 있게 되었습니다.

### 🌉 마침내 완성된 피드백 브릿지

1장과 2장에서는 무한 루프와 엉망진창인 코드 생성(지능의 실패)을 통제했고, 3장에서는 좀비 프로세스(OS의 실패)를 소탕했으며, 이번 4장에서는 텔레그램 데드락(네트워크의 실패)까지 완벽하게 극복해 냈습니다.

이제 제 Mac에 구축된 자율 개발팀은 그 어떤 외부 충격에도 멈추지 않는 불사신이 되었습니다.

이 모든 고난의 SRE 트러블슈팅을 거쳐 마침내 완성된 **'텔레그램 기반 완전 자율 무인 개발 루프'**가 실제로 어떻게 작동하는지, 대망의 5장(최종회)에서 그 시연과 워크플로우를 공개하겠습니다.

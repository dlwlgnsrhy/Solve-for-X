# TS-0007: Life-Log V4 omo(Gemma) 치명적 문법 오타 및 중단 보고서

## 📅 날짜: 2026-04-16
## 🏷️ 태그: #문법오류 #Gemma-4bit #omo #실패분석 #V4중단

---

## 1. 🚨 사건 요약
V3 환각을 방지하고자 V4 진행 중, omo(Gemma 31B 4-bit)가 1단계인 Entity 생성 단계에서부터 타입과 구문을 붕괴시키는 치명적 오타(`Map<<StringString, dynamic>`)를 발생시켰습니다. 해당 문법 오류를 복구하기보다 깨끗하게 V5로 새로 시작하는 것이 더 안전하다고 판단되어 작업을 중단했습니다.

## 2. 🔍 상세 진단 결과
### A. 양자화 모델 특유의 문법 붕괴 (Syntax Collapse)
- **현상**: `lib/domain/entities/checkin_data.dart` 내부에 `Map<<StringString, dynamic>` 와 같이 부등호 중복, 띄어쓰기 생략, 타입 선언 붕괴 발생.
- **원인**: 4-bit 모델 특성 상 구조화된 플러터(Dart) 문법 생성 시 정확도(Precision)가 순간적으로 떨어져 문장이 깨지는(말더듬) 현상이 일어남.
- **비고**: V3 때 나왔던 "반복적인 루프 오타" 현상과 궤를 같이 함.

## 3. 📉 영향 및 학습
- **초기 방역의 중요성**: Base Entity 코드에 한 줄이라도 문법적 오류가 있으면, 이후 이어지는 API, Provider 등 모든 코드가 연쇄적으로 망가집니다.
- **사람의 검증 필요성**: omo의 자동화 능력은 훌륭하지만, 각 파일이 생성될 때 *최소한 문법 에러(flutter analyze)는 패스했는지* 확실하게 체크하는 방어 지시어를 프롬프트에 넣어야 합니다.

## 4. 🛠️ 조치 및 향후 방침 (V5)
- **V5 클린 리뉴얼**: 구 버전의 영향을 완전히 배제하기 위해 `apps/life_log_v5` 디렉토리를 신규로 구성.
- **명령 체계 고도화**:
  1. 목표를 세분화하여 **계획 파일(blueprint)**을 별도의 마크다운으로 먼저 작성하게 함. (Prometheus 모형 등 활용)
  2. 작성된 청사진을 기반으로 `/start-work` 와 `/ulw-loop`를 이용한 심야(Night Shift) 자동화 릴레이를 돌릴 수 있도록 지시어 재설계.
- **아침 검증 가이드 도출**: 작업이 완료되었더라도, 아침에 일어나면 바로 `flutter test`와 `flutter analyze`로 컴파일 무결성부터 확인하는 절차 수립.

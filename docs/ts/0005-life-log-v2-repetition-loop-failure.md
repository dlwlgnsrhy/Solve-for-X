# 5. 로컬 양자화 모델(Gemma 4-bit)의 토큰 출력 한계 및 반복 루프 붕괴 트러블슈팅

Date: 2026-04-15

## Problem
1. **반복 루프 붕괴(Repetition Loop)**: omo(`Sisyphus` 에이전트)에 `apps/life_log_v2_premium` 전체 생성을 한 번의 프롬프트로 지시했으나, 첫 번째 파일인 `checkin_data.dart`를 작성하던 중 닫는 괄호 `}`가 450줄 이상 무한 반복되며 생성이 중단됨.
2. **후속 파일 미생성**: 앱 구조에 필요한 주요 UI 및 상태 관리, 네트워크 클라이언트 파일들이 전혀 생성되지 않은 상태로 프로세스가 조기 종료됨.

## Cause
- omo 내에서 사용 중인 모델은 `gemma-4-31B-it-AWQ-4bit`로 4-bit 양자화 모델임.
- 양자화된 로컬 모델은 한 번의 응답으로 긴 코드(수백 줄 이상의 묶음 파일)를 생성할 때 후반부에 접어들수록 모델의 컨텍스트 추론 능력이 급격히 저하됨.
- 방대한 작업을 단일 지시로 처리하려다 보니, 모델이 토큰 생성 한계를 견디지 못하고 "아무 의미 없는 문자의 무한 반복"이라는 생성형 AI 특유의 붕괴(Degradation) 패턴에 빠짐.

## Solution
1. **작업 세분화(Atomization)**: 한 번의 루프에서 전체 앱을 생성하려는 시도를 중단.
2. **파일 단위 릴레이 생성**: 전체 아키텍처를 사전에 설계하고, omo 에이전트에게 **정확히 1개의 파일**만 구현하라는 명시적인 명령을 연속적으로 전달.
3. **지속적 검증**: 각 파일이 생성될 때마다 `flutter analyze`와 같은 린트로 에러가 없음을 확인한 뒤, 그 상태와 컨텍스트(Session ID)를 바탕으로 다음 파일을 생성하는 '원자적 파이프라인' 도입.
4. **결과 보존 및 V3 세팅**: 터진 코드는 백업(`v2_failed`) 처리하고 새로운 실험용 워크스페이스 `life_log_v3_stepbystep` 구축.

## Note
- 이번 실패는 프롬프트의 질적 문제가 아니라 모델의 물리적인 '글자수(토큰) 출력 한계'에서 기인함.
- "설계자(관제탑)는 큰 그림을 그리고, 작은 작업만 작업자(omo)에게 지시한다"는 듀얼 에이전트의 원칙을 더 엄격하게 적용할 필요가 있음.

# 6. 바이오(수면) 데이터 추출 자동화 및 파이프라인 진화 전략

Date: 2026-04-14

## Status
Accepted

## Context
- '레거시 설계자'의 신체적, 정신적 컨디션을 파악하여 AI가 Daily Planner를 최적화하기 위해 스마트 링(삼성 헬스)의 수면 점수 데이터를 Mac 로컬 서버로 매일 아침 반입해야 하는 과제가 주어졌습니다.
- 삼성 헬스의 수면 데이터는 푸시 알림(Status Bar Notification)으로 제공되지 않으며, 앱 내부 동기화 시 조용히 갱신되는 특성이 있어 기존의 '알림 가로채기(Notification Interception)' 방식의 매크로가 작동하지 않는 문제가 발생했습니다.
- 외부 파이썬 스크립트가 클라우드에서 직접 로그인하여 삼성 헬스 데이터를 추출하는 것은 강력한 보안 정책 및 프라이빗 데이터 제약으로 인해 불가능합니다.
- 이로 인해 안드로이드 스마트폰 장치 내부에서 데이터를 중간 탈취(Scraping)해야 하지만, SRE/SWE 엔지니어링 관점에서 볼 때 스마트폰 매크로(MacroDroid 등)를 이용해 억지로 화면을 스크래핑하거나 버튼을 누르는 물리적 방식은 '진정한 서버 기반 업무 자동화'라 부르기 미흡한 부분이 있었습니다.

## Decision
- **단기/임시 전략 (Phase 1): MacroDroid 화면 스크래핑(위젯 버튼) 채택**
  - 현재로서는 시스템 파이프라인의 생애주기 테스트(PoC)를 가장 빠르게 증명하기 위한 수단으로 매크로드로이드의 화면 읽기 기능을 사용합니다.
  - 완전 무인화의 실패를 인정하고 1-Click(위젯 터치) 방식을 허용하되, Mac 쪽 서버 수신부(`health_receiver`)는 향후 어떤 형식의 데이터가 와도 정규식으로 안전하게 파싱하도록 고도화된 상태를 유지합니다.
- **궁극적 진화 전략 (Phase 3): 자체 Life-Log 앱 + Android Health Connect API 정규 연동**
  - 진정한 'Zero-Touch' 자동화를 달성하기 위해, 차기 로드맵의 핵심인 자체 Flutter 기반 `Life-Log 앱`에 구글 공식 `Health Connect API` 사용 권한을 획득합니다.
  - 지훈님의 기상 직후, 백그라운드 프로세스가 OS 단에서 수면 데이터를 꺼내어 Mac 봇 서버로 HTTP POST 요청을 쏘게 만드는 SRE급 아키텍처로 전면 교체(Migration)될 예정입니다.

## Consequences
- **Positive:**
  - MacroDroid를 임시로 사용함으로써 Health Receiver(Webhook) 기반의 AI 기동 아키텍처(Event-Driven) 자체는 빠르게 정립되었습니다.
  - "보이는 화면을 긁는 툴"과 "서버 간의 API 통신"의 구조적 한계와 명확한 방향성을 분별하고 차기 로드맵 목표의 가치를 격상하는 모티베이션이 되었습니다.
- **Negative:**
  - 자체 Life-Log 앱이 만들어지기 전(Phase 3 도래 전)까지는 수동으로 위젯 버튼을 누르거나, 화면 UI가 바뀔 때 매크로가 오작동할 수 있는 취약한 파이프라인 유지보수 비용이 발생합니다.
